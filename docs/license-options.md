# License Options

This is not legal advice. Public licensing is a human decision because it controls what other people may do with the project.

## Option A: MIT

Best when the priority is maximum developer adoption and simple reuse.

Pros:

- familiar to developers,
- permissive,
- works well if scripts or code are added later.

Cons:

- less tailored to prose/documentation than Creative Commons licenses.

## Option B: CC BY 4.0

Best when the project is mainly text, documentation, and instructional material.

Pros:

- natural fit for written material,
- permits reuse with attribution.

Cons:

- less conventional for code repositories if scripts are added later.

## Option C: Apache-2.0

Best when patent language matters or the project may grow into software.

Pros:

- permissive,
- includes explicit patent terms.

Cons:

- heavier than needed for an instruction-only Skill.

## Option D: No License Yet

Best when you want public visibility but are not ready to grant reuse rights.

Pros:

- avoids accidental permission grant.

Cons:

- limits legitimate reuse and contributions.

## Practical Recommendation

For a text-first Skill whose goal is broad adoption, choose either:

- `CC BY 4.0` if you want attribution to matter,
- `MIT` if you want the most developer-friendly default.

Do not publish a final `LICENSE` file until the owner explicitly chooses.
